# PMS / RMS: проблемы из чата TL → гипотезы фич → метрики успеха

Источник контекста: выборочный анализ `TL_llm_2025-04-08_2026-04-08.md` (Telegram, TravelLine Discuss).

| Проблема (как у пользователей) | Гипотеза фичи | Метрика успеха |
|--------------------------------|---------------|----------------|
| Нет полноценного открытого API для поиска доступности, создания брони и актуальных тарифов; автоматизация (боты, ИИ, сторонние PMS) упирается в закрытость или обходные пути | Публичный **Developer Portal**: REST/GraphQL (или аналог) для availability search, create/update booking, read tariffs & restrictions; песочница, ключи на объект, версионирование, webhooks по изменению брони/цены | Доля объектов с ≥1 активной интеграцией по API; время до первой успешной тестовой брони (TTV); снижение обращений в поддержку по теме «интеграция/выгрузка»; NPS/CSAT сегмента «разработчики/сети» |
| Данные из каналов (ОТА) искажают учёт и RMS: нет связки тариф–питание, средняя цена по заявке вместо разбивки по дням; расхождение статуса оплаты в PMS и фактических выплат/актов; споры по комиссиям; влияние на ADR/RevPAR | **Нормализация канала**: импорт цены по ночам; явная маппина meal/rate plan; флаги «оплачено у агента» vs «деньги у отеля»; сверка с агентским актом; предупреждения при аномалиях комиссии | % броней с полной дневной разбивкой цены без ручной правки; среднее время на сверку канала (опрос/сессии); число инцидентов «сумма в PMS ≠ акт»; стабильность отчётных ADR/RevPAR до/после смены интеграции канала |
| Тарифная модель не закрывает долгие брони, почасовые залы в одной шахматке, маркеры оплат по долгосроку; MinLos/availability путают пользователей; ранний заезд/поздний выезд при тарифе с питанием считаются не от той базы — массовый ручной пересчёт | **Гибкий шаг тарификации** (сутки / месяц / час) и привязка к типу ресурса; **платёжные якоря** на таймлайне брони; настройка базы для РЗ/ПВ (от «ночи без питания» / от полного тарифа); **массовые операции** с ценами (копирование периода + % с защитой уже изменённых дат) | Снижение доли броней с ручной правкой суммы после создания; время массового обновления цен на сезон (до/после); CSAT по модулю тарифов; доля обращений «MinLos / РЗ-ПВ / долгосрок» |
| Разрыв между налоговой/бух отчётностью и операционной логикой: турналог и смежные отчёты требуют ручной правки сотен строк, сверка TL ↔ 1С, разные срезы (оплата / заезд / проживание) | Два согласованных среза в продукте: **«регуляторика»** (по правилам закона) и **«управление»** (по заезду/ночам); настраиваемые колонки; выгрузки без потери связи с бронью; API/фид для 1С с тем же определением периода | Доля пользователей, закрывающих период без выгрузки в Excel; время закрытия месяца (опрос); число тикетов «ошибка в отчёте / турналог»; совпадение итогов TL ↔ 1С на пилотных объектах (контрольная выборка) |
| Операционный UX и масштаб: нет группировки шахматки по корпусам; деградация браузера при долгой работе; мобильная лента с задержками и регрессом UX; сетям не хватает сквозной аналитики; большие группы при малом числе ПК заселения | **Группировка/фильтры** шахматки по корпусу/зданию; профилирование и снижение утечек на фронте; **SLA ленты** броней в app; **сетевая аналитика** (консолидация, drill-down); сценарии массового заселения / очереди задач | Lighthouse/Web Vitals на длительной сессии; crash rate / OOM в браузере (обратная связь); p95 задержки «событие в канале → пуш/лента»; adoption сетевых дашбордов; время заселения группы (пилот) |

## Дополнительно: продуктовая аналитика (сквозные гипотезы)

| Направление | Гипотеза фичи | Метрика успеха |
|-------------|---------------|----------------|
| Каналы и RMS | Единая **событийная модель** цены (по ночам, нетто/вал, комиссия, пакет) + дашборд диспаритета и pickup | Корреляция отчётного RevPAR с контрольной выгрузкой; время обнаружения расхождения с ОТА |
| Атрибуция | Явные правила **источника брони** и согласованность с внешними счётчиками (метрика, виджеты не на сайте) | Доля броней с однозначным источником; расхождение «метрика vs PMS» на тестовых кампаниях |
| Ценообразование | **Аудит-лог** изменений тарифов, ограничений и цен (кто, когда, было/стало) | Доля вопросов «почему цена изменилась» с закрытием без поддержки; время расследования инцидента |
| Надёжность RMS | Мониторинг доступности оптимизатора/динамики как **SLO** с статус-страницей | Uptime целевого модуля; MTTR; число жалоб «не открывается раздел» |
