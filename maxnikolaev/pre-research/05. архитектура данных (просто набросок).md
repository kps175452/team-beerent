# Data architecture: AI/data layer for hospitality

Дата: 28 марта 2026

## 1. Принцип архитектуры

Продукт не должен требовать полной миграции клиента на новую PMS. Архитектура строится как `overlay decision layer`, который:
- подключается к уже существующим системам;
- регулярно получает данные;
- обогащает их внешними источниками;
- рассчитывает рекомендации;
- возвращает действия в интерфейс, отчеты, экспорт или интеграции.

Идея проста:

`данные -> нормализация -> enrichment -> decision engine -> рекомендации -> действия`

## 2. Основные блоки архитектуры

### 1. Data ingestion layer

Этот слой собирает данные из внутренних и внешних источников.

Форматы входа:
- API-интеграции;
- регулярные CSV/Excel-выгрузки;
- web scraping / парсинг открытых данных;
- ручная загрузка файлов;
- webhook / event feed;
- scheduled polling.

### 2. Normalization layer

Так как данные приходят из разных систем и в разном качестве, их нужно привести к единой модели:
- единые идентификаторы объектов;
- единые справочники услуг;
- единая модель бронирования;
- единая модель гостя;
- единая модель цен и загрузки;
- единая модель внешних событий и конкурентного окружения.

### 3. Enrichment layer

После нормализации данные дополняются новыми признаками:
- сезонность;
- тип дня;
- proximity to events;
- сегмент спроса;
- ценовая позиция относительно конкурентов;
- вероятность upsell;
- склонность к определенному набору услуг;
- channel mix;
- lead time;
- length of stay.

### 4. Decision engine

Сердце продукта. Здесь работают:
- forecasting;
- pricing models;
- rules engine;
- cohorting / segmentation;
- recommendation logic;
- LLM-based explanation layer;
- alerting logic.

### 5. Delivery layer

Результаты должны попадать в бизнес в прикладной форме:
- dashboard;
- email / Telegram / Slack alerts;
- PDF/Excel reports;
- экспорт рекомендаций обратно в PMS/CRM;
- staff-facing copilot;
- API для enterprise clients.

## 3. Источники данных

## A. Внутренние данные клиента

### PMS / ERP / booking data

Что берем:
- бронирования;
- отмены;
- no-show;
- заезды/выезды;
- загрузка;
- тарифы;
- room type mix;
- длина проживания;
- канал бронирования.

Зачем:
- базовая модель спроса;
- расчет occupancy, ADR, RevPAR;
- forecasting;
- анализ поведения по датам и сегментам.

### CRM / guest history

Что берем:
- история гостей;
- повторные визиты;
- предпочтения;
- история коммуникаций;
- прошлые покупки услуг.

Зачем:
- guest segmentation;
- retention;
- персонализированные рекомендации;
- propensity scoring для upsell.

### Ancillary sales / internal service catalog

Что берем:
- spa;
- баня;
- трансферы;
- экскурсии;
- аренда оборудования;
- питание;
- праздничные пакеты;
- upgrade inventory.

Зачем:
- ancillary revenue optimization;
- подбор релевантных предложений;
- построение service recommendation engine.

### Marketing / direct channel data

Что берем:
- direct bookings;
- конверсия сайта;
- abandoned bookings;
- response на офферы;
- источники трафика.

Зачем:
- direct revenue analytics;
- campaign optimization;
- demand shaping.

## B. Внешние источники данных

### Competitor pricing

Примеры:
- OTA / публичные источники;
- агрегаторы;
- marketplaces;
- Avito Путешествия;
- собственный comp set parser.

Зачем:
- rate positioning;
- gap analysis;
- price response;
- мониторинг недооцененных дат.

### Demand/event calendar

Примеры:
- конференции;
- форумы;
- концерты;
- фестивали;
- локальные события;
- школьные каникулы;
- праздничные периоды.

Зачем:
- event-driven pricing;
- occupancy forecasting;
- early alerts по росту спроса.

### Regional tourism and macro signals

Примеры:
- турпоток;
- сезонность направления;
- транспортная доступность;
- погодные факторы;
- длинные выходные;
- авиасообщение.

Зачем:
- более устойчивая макромодель спроса;
- объяснение аномалий;
- более качественные forecast features.

### Open-web guest context

Этот блок нужно использовать очень осторожно.

Возможные данные:
- публично предоставленные данные самим гостем;
- добровольно собранные preference data;
- результаты анкетирования;
- first-party behavioral data.

Важно:
- не строить критическую часть продукта на сомнительных практиках с персональными данными;
- разделять допустимое enrichment и risky profiling;
- сначала опираться на first-party data.

## 5. Что именно считает decision engine

### 1. Pricing recommendations

Вопрос:
- какую цену ставить на конкретную дату и room type?

Входы:
- historical bookings;
- pace;
- competitor rates;
- events;
- seasonality;
- occupancy forecast.

Выход:
- recommended rate;
- delta from current rate;
- confidence score;
- explanation.

### 2. Demand forecasting

Вопрос:
- что будет с загрузкой и спросом на горизонте 7/14/30/90 дней?

Выход:
- occupancy forecast;
- expected pickup;
- alert on unusual demand.

### 3. Ancillary recommendation

Вопрос:
- что и кому продавать дополнительно?

Входы:
- профиль гостя;
- тип поездки;
- stay pattern;
- состав услуг объекта;
- история конверсии по похожим когортам.

Выход:
- ranked list of offers;
- timing of offer;
- channel of offer;
- staff instruction.

### 4. Staff copilot prompt

Вопрос:
- что должен знать и сделать сотрудник по этому гостю или заезду?

Выход:
- краткая инструкция;
- чек-лист действий;
- фразы и рекомендации;
- приоритеты.

### 5. Benchmarking

Вопрос:
- где объект теряет деньги относительно рынка?

Выход:
- market position score;
- underpriced dates;
- missed ancillary opportunities;
- segment gap analysis.

## 6. Архитектура развития по этапам

## Stage 1. MVP

Цель:
- доказать деньги на pricing.

Источники:
- PMS exports;
- собственные цены;
- competitor rates;
- event calendar.

Выход:
- pricing dashboard;
- daily recommendations;
- simple benchmark.

## Stage 2. Commercial intelligence

Добавляем:
- channel mix;
- CRM data;
- direct booking analytics;
- ancillary catalog.

Выход:
- pricing + offer strategy;
- basic guest segmentation;
- direct revenue insights.

## Stage 3. Guest/staff intelligence

Добавляем:
- более глубокие first-party guest signals;
- communication data;
- repeat stay patterns;
- offer response history.

Выход:
- guest-specific recommendations;
- staff copilot;
- retention logic.

## Stage 4. Enterprise layer

Добавляем:
- multi-property architecture;
- API back into client systems;
- role-based workflows;
- approvals and audit logs;
- group benchmarking.

Выход:
- scalable platform for chains and operators.

## 7. Как запускать без тяжелых интеграций

Практический порядок:

1. Начать с CSV/API экспорта из PMS.
2. Подключить внешние цены и события.
3. Доказать пользу на отчетах и рекомендациях.
4. Только потом делать глубокие интеграции и real-time loop.

Это критично, потому что:
- снижает friction на старте;
- ускоряет time-to-value;
- позволяет заходить даже в объекты с несовершенным стеком.

## 8. Основные продуктовые риски

### Data quality risk

Если внутренние данные клиента грязные, модели будут шумными.

Нужен слой:
- validation;
- anomaly detection;
- fallback rules.

### Integration risk

Слишком ранняя ставка на глубокую интеграцию замедлит go-to-market.

### Privacy/compliance risk

Нельзя делать risky profiling центром MVP.

Безопасная стратегия:
- first-party data first;
- внешние персональные данные только там, где есть понятная легальная и продуктовая рамка.

### Overbuild risk

Если пытаться с первого дня построить “полный мозг для всей отрасли”, проект потеряет фокус.

## 9. Рекомендуемая архитектурная формула старта

Лучший старт:

`PMS export + competitor parser + event signals + recommendation dashboard`

Затем:

`CRM + ancillary catalog + staff copilot`

Потом:

`enterprise integrations + multi-property intelligence`

## 10. Операционная модель архитектуры

Очень важно зафиксировать: архитектура строится не как “вторая PMS, в которой дублируется вся жизнь объекта”.

Она строится как 3-слойная модель:

- `system of record`
- `system of intelligence`
- `system of action`

### System of record

Это текущая PMS / ERM клиента, если речь идет о зрелом объекте.

Там живут:
- первичные бронирования;
- operational статусы;
- транзакционный учет;
- основной поток жизни объекта.

### System of intelligence

Это Beerent.

Здесь живут:
- каноническая модель;
- цифровой двойник объекта;
- внешние сигналы спроса;
- benchmark;
- pricing / demand / ancillary logic;
- объяснение решений.

### System of action

На ранней стадии это:
- approval center;
- экспорт;
- staff brief;
- digest.

На зрелой стадии это:
- selective write-back;
- частичная автоматизация команд обратно в PMS;
- shell mode для отдельных клиентов.

## 11. Что такое digital twin в контексте Beerent

Digital twin — это не вторая вручную поддерживаемая шахматка.

Это автоматически обновляемое представление объекта, собранное из:
- PMS exports;
- API feeds;
- scheduled imports;
- внешних источников данных.

### Что twin должен содержать

- объекты и их структуру;
- room types / units;
- тарифы;
- бронирования;
- загрузку;
- guest history;
- каталог услуг;
- источники бронирований;
- pricing snapshots;
- external signals;
- рекомендации.

### Что twin не должен требовать

- ручного заведения бронирований второй раз;
- параллельного ведения номерного фонда;
- ручной поддержки календаря для mature-клиента;
- постоянной работы персонала в двух системах.

## 12. Customer Journey Map на уровне архитектуры данных

Ниже важен не маркетинговый, а технический путь данных.

## Stage 1. Pilot intake

Клиент:
- выбирает один объект;
- дает выгрузку или API-доступ;
- подтверждает набор необходимых сущностей.

Beerent:
- регистрирует integration config;
- создает property workspace;
- запускает первичный ingestion.

## Stage 2. Initial backfill

Beerent:
- принимает бронирования;
- принимает тарифы;
- принимает room inventory;
- принимает список услуг;
- приводит все в каноническую модель.

Результат:
- первый digital twin;
- базовый benchmark;
- readiness score по данным.

## Stage 3. Ongoing sync

Дальше twin должен жить не вручную, а через один из sync-режимов:

### Sync mode A. Manual upload

Подходит для очень ранних пилотов.

### Sync mode B. Scheduled export

Подходит для большинства первых рабочих внедрений.

### Sync mode C. API / webhook

Подходит для зрелых клиентов и select enterprise accounts.

## Stage 4. Recommendation loop

После обновления twin Beerent:
- пересчитывает признаки;
- обновляет pricing model;
- считает benchmark gap;
- строит offer suggestions;
- формирует staff / management outputs.

## Stage 5. Action loop

На раннем этапе:
- approval;
- export;
- manual apply в PMS.

На зрелом этапе:
- price write-back;
- note / task write-back;
- workflow automation.

## 13. Где действительно возникает ручная работа, а где не должна возникать

Это нужно фиксировать архитектурно, чтобы не скатиться в неверную реализацию.

### Допустимая ручная работа

На пилоте допустимо:
- один раз настроить mapping полей;
- запускать или контролировать регулярную выгрузку;
- вручную применять часть ценовых рекомендаций;
- сверять benchmark и результаты.

### Недопустимая ручная работа

Нельзя проектировать систему так, чтобы клиент:
- дублировал каждую бронь;
- руками поддерживал twin;
- вел два operational календаря;
- переводил статусы проживания в двух местах;
- использовал Beerent как вторую обязательную PMS для mature-сценария.

## 14. Почему overlay-архитектура не означает слабый продукт

Наоборот, для зрелых клиентов это часто более правильная архитектура.

Потому что:
- system of record уже существует;
- менять ее дорого;
- ценность часто лежит не в учете, а в intelligence-слое;
- на enterprise-рынке принятие решений и automation поверх существующего стека часто ценнее, чем forced migration.

### Архитектурная формула для enterprise

`current PMS = system of record`

`Beerent = system of intelligence`

`write-back / shell / approval = system of action`

## 15. Когда нужна миграция, а когда не нужна

Архитектурно это должен быть `decision tree`, а не догма.

### Сценарий A. Mature client

Обычно остается на overlay-модели.

Почему:
- процессы уже выстроены;
- risk of migration слишком дорогой;
- Beerent получает value без переезда.

### Сценарий B. Greenfield client

Может использовать Beerent shell как основную operational оболочку.

### Сценарий C. Hybrid client

Может:
- часть объектов держать на старой PMS;
- новые объекты запускать на Beerent shell;
- постепенно унифицировать стек, если это реально выгодно.

Архитектура должна поддерживать все 3 сценария.

## 16. Какие технические модули архитектуры нужно выделить в первую очередь

### 1. Connector registry

Хранит:
- тип источника;
- credentials;
- sync policy;
- mapping profile;
- freshness state.

### 2. Import center

Отвечает за:
- ручную загрузку;
- шаблоны файлов;
- валидацию;
- пользовательский onboarding данных.

### 3. Schema mapper

Отвечает за:
- сопоставление полей;
- трансформации;
- fallback rules;
- сохранение mapping templates.

### 4. Sync orchestrator

Отвечает за:
- backfill;
- delta updates;
- ретраи;
- conflict handling;
- freshness SLAs.

### 5. Twin store

Это материализованное представление объекта для продукта Beerent.

### 6. Recommendation engine

Ядро intelligence-слоя.

### 7. Approval / export layer

Превращает intelligence в действия.

### 8. Write-back adapter layer

Нужен позже, но должен быть предусмотрен архитектурно с самого начала.

## 17. Приоритет write-back по сущностям

Нельзя пытаться писать обратно все подряд.

Правильный порядок такой:

### Priority 1

- цены;
- rate plans;
- ограничения по датам, если это поддерживается.

### Priority 2

- notes;
- simple staff tasks;
- теги / флаги.

### Priority 3

- offer triggers;
- часть guest communication events;
- более сложные workflow commands.

Операционные транзакции core PMS не должны быть первым объектом write-back.

## 18. Архитектурная формула MVP

Если предельно приземленно, MVP архитектуры должен отвечать на один вопрос:

`как быстро построить цифрового двойника, показать money insight и не заставить клиента жить в двух системах`

Поэтому MVP архитектурно — это:

- import center;
- canonical model;
- twin store;
- competitor / event enrichment;
- pricing recommendation engine;
- cockpit;
- approval + export.

## 19. Что это меняет в архитектурном приоритете команды

Команда должна тратить основные усилия не на:
- расширение parity PMS-функций;
- операционный edge-case coverage;
- бесконечное доращивание шахматки.

А на:
- ingestion reliability;
- data quality;
- twin freshness;
- explainability of recommendations;
- ROI attribution;
- selective integration depth.

## 20. Финальная архитектурная позиция

Beerent архитектурно должен восприниматься так:

`не как альтернативная транзакционная система для всех клиентов, а как data-driven intelligence layer с собственным shell, который при необходимости может стать action layer для части сегментов`

Это и позволяет:
- не выкинуть уже сделанную шахматку;
- не уйти в слабую стратегию “еще одна PMS”;
- и при этом сохранить технически цельную продуктовую линию.
